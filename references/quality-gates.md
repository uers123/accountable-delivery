# 质量门禁定义

> 质量门禁是交付物必须通过的检查关卡。每个门禁都有明确的通过标准。
> 根据任务类型选择合适的门禁组合，不可跳过硬性门禁。

---

## 硬性门禁（所有任务必须通过）

### Q1: 功能完整性
| 项目 | 内容 |
|------|------|
| **目的** | 确保所有需求中明确的功能都已实现 |
| **检查方式** | 对照需求列表逐项检查，每个功能点有对应的测试用例 |
| **通过标准** | 100% 的功能点已实现并通过测试 |
| **失败示例** | "需求说支持 3 种导出格式，只实现了 2 种" |

### Q2: 结果正确性
| 项目 | 内容 |
|------|------|
| **目的** | 确保输出与预期一致，计算逻辑正确 |
| **检查方式** | 断言验证、预期输出对比、已知样本测试 |
| **通过标准** | 所有断言通过，输出与预期完全一致 |
| **失败示例** | "输入 'Hello' 期望输出 'olleH'，实际输出 'Hel lo'" |

### Q3: 异常处理
| 项目 | 内容 |
|------|------|
| **目的** | 确保程序在非法输入或异常情况下不崩溃，给出有意义的错误信息 |
| **检查方式** | 空值测试、边界值测试、非法格式测试、资源不可用测试 |
| **通过标准** | 所有异常路径被捕获，程序不崩溃，错误信息可理解 |
| **失败示例** | "传入 None 时抛出 AttributeError 而不是返回友好的错误信息" |

### Q4: 无退化
| 项目 | 内容 |
|------|------|
| **目的** | 确保新代码没有破坏已有功能 |
| **检查方式** | 回归测试——运行全部已有测试用例 |
| **通过标准** | 所有已有测试用例仍然通过 |
| **失败示例** | "修复了 Bug A，但原来正常的功能 B 现在出错了" |

## 建议门禁（根据任务类型选择）

### Q5: 性能基线
| 项目 | 内容 |
|------|------|
| **目的** | 确保程序在可接受的性能范围内运行 |
| **检查方式** | 计时测试、资源监控（内存/CPU） |
| **通过标准** | 响应时间 < 预期阈值，无内存泄漏迹象 |
| **适用场景** | 数据处理、批量操作、实时接口 |
| **失败示例** | "处理 1000 条记录需要 5 分钟，预期应 < 10 秒" |

### Q6: 代码可读性
| 项目 | 内容 |
|------|------|
| **目的** | 确保代码清晰可维护，他人能看懂 |
| **检查方式** | 自我审查：命名、结构、注释、类型注解 |
| **通过标准** | 函数/变量命名自解释，有类型注解，复杂逻辑有注释，无死代码 |
| **适用场景** | 所有生产代码 |
| **失败示例** | "变量名 a、b、c，无类型注解，300 行的函数无注释" |

### Q7: 输入验证
| 项目 | 内容 |
|------|------|
| **目的** | 确保程序能正确处理各种合法和非法的输入 |
| **检查方式** | 类型检查、范围检查、格式验证 |
| **通过标准** | 所有输入参数经验证后才使用，非法输入被拒绝并给出提示 |
| **适用场景** | 用户输入处理、API 接口、文件处理 |
| **失败示例** | "传入了负数作为数组索引，导致未定义行为" |

### Q8: 安全性
| 项目 | 内容 |
|------|------|
| **目的** | 确保代码不存在明显安全漏洞 |
| **检查方式** | 注入风险检查、敏感数据处理检查、路径遍历检查 |
| **通过标准** | 不存在命令注入、路径遍历、硬编码凭据等常见安全问题 |
| **适用场景** | Web 服务、文件操作、用户数据处理 |

### Q9: 兼容性
| 项目 | 内容 |
|------|------|
| **目的** | 确保代码在目标环境中正常运行 |
| **检查方式** | 版本依赖检查、平台差异测试 |
| **通过标准** | 在目标 Python/Node.js 版本上运行正常，依赖版本兼容 |
| **适用场景** | 需要部署到特定环境的代码 |

## 门禁选择指南

| 任务类型 | 必选门禁 | 建议门禁 |
|---------|---------|---------|
| 新增功能 | Q1, Q2, Q3, Q4 | Q6 代码可读性 |
| Bug 修复 | Q2, Q3, Q4 | Q6 代码可读性 |
| 性能优化 | Q4, Q5 | Q6 代码可读性 |
| 重构 | Q2, Q4, Q6 | Q5 性能基线 |
| API 开发 | Q1, Q2, Q3, Q4, Q6 | Q7 输入验证, Q8 安全性 |
| 数据处理脚本 | Q1, Q2, Q3, Q4 | Q5 性能基线 |
| Web 应用 | Q1, Q2, Q3, Q4, Q6 | Q7, Q8, Q9 |
| 命令行工具 | Q1, Q2, Q3, Q4, Q6 | Q7 输入验证 |

## 门禁未通过的处理流程

```
门禁未通过
    ↓
记录哪个门禁失败以及原因
    ↓
进入反思阶段（SKILL.md 阶段 3）
    ↓
分类错误类型
    ↓
分析根因 → 记录教训 → 制定纠正方案
    ↓
修改代码
    ↓
重新验证该门禁 → [通过] 检查其他门禁 → [全部通过] 交付
                    → [仍失败] 再次反思
```
