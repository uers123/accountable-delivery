# 自我问责交付流程

> 核心原则：我对这个交付物负责。不测试不交付，不达标不提交，不反思不进步。

---

## 阶段 0：启动声明

开始任何交付任务前，必须先声明以下承诺：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  自我问责声明
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ✅ 我对这个交付物负全部责任
  ✅ 我先测试、后交付
  ✅ 测试不通过则反思根因并返工
  ✅ 全部通过质量门禁才提交
  ✅ 每一次失败都是进化的机会
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## 阶段 1：需求确认与质量门禁定义

在写任何代码之前，必须：

### 1.1 复述需求
```
我用我的语言复述需求：
  需求目标：XXX
  输入是什么：XXX
  输出是什么：XXX
  关键约束：XXX
请确认我的理解是否正确。
```

### 1.2 定义质量门禁
根据任务性质，从 `references/quality-gates.md` 中选择适用的门禁，明确每个门禁的**通过标准**。

示例：
| 门禁 | 通过标准 |
|------|---------|
| 功能完整性 | 全部 5 个功能点均已实现 |
| 结果正确性 | 测试用例 100% 通过 |
| 异常处理 | 空值、边界值不崩溃 |
| 代码可读性 | 命名清晰、有类型注解、有关键注释 |

---

## 阶段 2：实现与自测

### 2.1 编写实现
遵循好的工程实践：清晰命名、单一职责、类型注解、错误处理。

### 2.2 立即编写并运行测试
- 针对每个功能点编写测试用例
- 覆盖正常路径、边界条件、异常输入
- 运行测试并记录结果

### 2.3 记录测试状态
```
测试记录：
  [✓] 测试用例 1：描述 - 通过
  [✗] 测试用例 2：描述 - 失败
  [✓] 测试用例 3：描述 - 通过
  总计：X 通过 / Y 失败
```

如果全部通过 → 跳至阶段 5
如果有失败 → 进入阶段 3

---

## 阶段 3：反思与根因分析（核心）

当测试失败时，**必须**进行结构化反思。这是本技能区别于普通测试的关键所在。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  结构化反思日志
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 失败描述
   什么测试失败了？期望什么？实际得到什么？

2. 失败分类（选择一种）
   □ 需求理解错误 —— 我理解的需求和用户实际需求不一致
   □ 技术实现错误 —— 我知道要做什么，但实现方式不对
   □ 边界遗漏 —— 没有考虑到某种输入或场景
   □ 性能/质量不达标 —— 功能对了但不够好

3. 根因分析
   为什么会发生这个错误？最根本的原因是什么？

4. 学到的教训
   以后如何避免同类错误？这个教训可以泛化到哪些场景？

5. 纠正方案
   具体怎么修改代码？修改后如何验证？

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

反思完成后，进入阶段 4。

---

## 阶段 4：返工与验证

### 4.1 执行纠正
根据阶段 3 的纠正方案修改代码。

### 4.2 重新测试
- 重新运行所有测试
- 如果仍有失败 → 返回阶段 3（再次反思）
- 如果全部通过 → 进入阶段 5

### 4.3 记录返工次数
```
返工记录：#X 次返工
  本轮错误类型：XXX
  本轮教训：XXX
```

---

## 阶段 5：交付与总结

### 5.1 最终验证
逐一核对阶段 1 定义的质量门禁：
```
质量门禁核验清单：
  [✓/✗] 功能完整性
  [✓/✗] 结果正确性
  [✓/✗] 异常处理
  [✓/✗] 性能基线
  [✓/✗] 代码可读性
  [✓/✗] 无退化
全部通过？[是/否]
```

### 5.2 交付总结报告

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  交付总结报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  需求：XXX
  质量门禁：全部 [✓] / 未全部 [✗]
  测试覆盖：XX 个测试用例，全部通过
  返工次数：X 次
  关键教训：
    1. XXX
    2. XXX
  本轮进化：XXX
  交付置信度：[高/中/低]
  交付决定：[可交付 / 不可交付]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 5.3 更新自我进化库

将本次学到的教训追加到进化记录中：
```
## 自我进化记录
#[N] [日期] 项目：[项目名]
  错误类型：[分类]
  教训：[具体教训]
  应用场景：[未来如何应用]
```

**只有全部质量门禁通过 → 才能交付。**

---

## 关键规则

1. **不可跳过阶段 3**：测试失败时不反思就返工，是低效的修补，不是真正的改进
2. **不可虚假通过**：测试必须真实运行并验证结果，不能假装测试通过
3. **教训必须记录**：不记录教训等于没有进化
4. **质量门禁不可妥协**：任何门禁未通过就不交付
5. **诚实报告置信度**：如果对交付物没有信心，必须如实报告"低"置信度并说明原因
